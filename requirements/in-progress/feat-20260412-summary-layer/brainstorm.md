# Brainstorm: summary 层

## 背景

当前 `context-builder` 已经预留了 `summary` section，但实际返回为空。也就是说，系统虽然有了摘要层的接入点，却还没有真正形成可复用的知识压缩层。

如果 summary 一直缺位，Thinking Tree 的上下文就会长期依赖原始 QAPair 和 Block。树一大，系统又会回到“历史堆叠”的模式，难以把提问过程沉淀成知识结构。

## 现状问题

### 1. summary 还只是占位

现在没有真正可消费的摘要内容，节点、路径、子树都还没有摘要层。

结果是：

- prompt 里缺少压缩后的高价值信息
- 非当前路径分支无法有效复用
- 树越长，越依赖原始历史，token 压力越大

### 2. 摘要生成时机还没落定

summary 可以在多个时机生成：

- 聊天结束后同步生成
- 聊天结束后异步生成
- 用户显式触发生成
- 读取上下文时按需生成

不同策略会影响交互延迟、实现复杂度和数据一致性。

### 3. 摘要粒度需要统一

summary 至少包含两类：

- 节点摘要：压缩某个节点的问答内容
- 子树摘要：压缩某个分支下的知识结论

我们决定先把这两类一起纳入同一条 pipeline，再放到 `collect` 阶段统一注册，后面一起看效果。

### 4. 摘要的消费方还不清晰

summary 最终可能被这些地方消费：

- `context-builder` 的 prompt
- tree 面板上的结构展示
- 未来的检索或相关性选择

如果不先区分消费方，很容易把摘要做成单一用途，后面再返工。

## 已确认方向

这次 summary pipeline 先按以下决策推进：

- summary 同时覆盖节点摘要和子树摘要
- 继续沿用 `collect -> select -> format` 的分层结构
- 先把 summary 注册进 `collect` 阶段
- summary 采用异步生成
- summary 完全由模型生成，暂不开放人工编辑
- summary 缺失时可以完全忽略，不在 prompt 里强行占位

## 目标

### 核心目标

1. 定义 summary 层在 Cognitree 中的职责边界。
2. 明确 summary 的粒度、生成时机和消费方式。
3. 为后续把 summary 接进 `context-builder`、界面和检索逻辑打好基础。

### 体验目标

1. 不破坏当前聊天链路。
2. 不要求前端立刻配合复杂改造。
3. 先让摘要层“能被使用”，再追求“摘要生成得多聪明”。

## 范围

### In Scope

- 设计 summary 的层级模型
- 确认 summary 的生成策略
- 确认 summary 的存储和读取路径
- 确认 summary 在 prompt 中的优先级
- 设计摘要缺失时的 fallback

### Out of Scope

- 一次性把所有摘要生成逻辑都实现完
- 前端大幅改版 summary UI
- 引入外部 RAG 系统
- 复杂的多模型摘要路由

## 设计方向

### 方向 1: 节点摘要优先

先做节点级 summary，把单个节点的问答压缩成简短结论。

优点：

- 最容易接入当前数据模型
- 可以立即服务于 context-builder

缺点：

- 对大分支的整体知识压缩能力有限

### 方向 2: 子树摘要优先

直接做某个分支下的树摘要，优先解决“知识沉淀”。

优点：

- 更符合 Thinking Tree 的知识树目标

缺点：

- 实现和维护复杂度更高

### 方向 3: 双层摘要

同时定义节点摘要和子树摘要，先落节点摘要，再逐步聚合成子树摘要。

优点：

- 最符合长期演进方向
- 便于后续分阶段落地

缺点：

- 设计阶段需要先把边界切清楚

## 建议的设计原则

1. summary 必须可消费，不能只是数据库里的空字段。
2. summary 必须有清晰粒度，不能混淆节点与子树职责。
3. summary 必须支持渐进落地，先可用，再变聪明。
4. summary 缺失时可以完全忽略，不额外占位。

## 开放问题

1. summary 是否先只做节点 + 子树双层，不再拆路径摘要？
2. summary 生成是否需要失败重试或补偿任务？
3. summary 是否保留人工编辑入口，还是完全由模型生成？
4. summary 缺失时，是否也要在日志里记录一次可观测事件？
