# Brainstorm: summary 层

## 背景

当前 `context-builder` 已经预留了 `summary` section，但实际返回为空。也就是说，系统已经有了摘要层的“接入点”，只是还没有真正产生可复用的内容。

如果 summary 继续缺位，Thinking Tree 的上下文只能长期依赖原始 QAPair 和 Block，树一大就会重新回到“历史堆叠”模式，无法形成可压缩、可复用的知识层。

## 现状问题

### 1. summary 只是占位，不是知识层

现在的摘要层没有真正承担任何职责，既没有节点摘要，也没有路径摘要，更没有子树摘要。

结果就是：

- prompt 里没有压缩后的高价值信息
- 非当前路径分支无法被有效复用
- 越长的树越依赖原始历史，token 压力越大

### 2. 摘要生成时机还没定义

当前还不清楚 summary 应该在什么时候生成：

- 聊天结束后同步生成
- 聊天结束后异步生成
- 用户显式触发生成
- 读取上下文时按需生成

不同策略会直接影响交互延迟、实现复杂度和数据一致性。

### 3. 摘要粒度还没切清楚

summary 至少可能有三种粒度：

- 节点摘要：压缩某个节点的问答内容
- 路径摘要：压缩根到当前节点的探索链路
- 子树摘要：压缩某个分支下的知识结论

这三种粒度的职责并不一样，不能混成一个字段草率处理。

### 4. 摘要的消费方不清晰

摘要最终可能被这些地方消费：

- `context-builder` 的 prompt
- tree 面板上的结构展示
- 未来的检索或相关性选择

如果一开始不区分消费方，很容易把摘要做成单一用途，后续再返工。

## 用户价值

1. 树变大之后，模型仍能通过摘要快速抓住关键知识。
2. 已经探索过的分支可以以更短的形式复用，而不是反复重放原始对话。
3. 树的历史会从“记录发生了什么”升级成“记录结论是什么”。

## 需求目标

### 核心目标

1. 定义 summary 层在 Cognitree 中的职责边界。
2. 明确 summary 的粒度、生成时机和消费方式。
3. 为后续把 summary 接进 `context-builder`、界面和检索逻辑打好基础。

### 体验目标

1. 不破坏当前聊天链路。
2. 不要求前端立即配合复杂改造。
3. 先让摘要层“能被使用”，再追求“摘要生成得多聪明”。

## 范围

### In Scope

- 设计 summary 的层级模型
- 确认 summary 的生成策略
- 确认 summary 的存储和读取路径
- 确认 summary 在 prompt 中的优先级
- 设计摘要缺失时的 fallback

### Out of Scope

- 一次性把所有摘要生成逻辑都实现完
- 前端大幅改版 summary UI
- 引入外部 RAG 系统
- 复杂的多模型摘要路由

## 关键决策方向

### 方向 1: 节点摘要优先

先做节点级 summary，把单个节点的问答压缩成简短结论。

优点：

- 最容易接入当前数据模型
- 可以立即服务于 context-builder

缺点：

- 对大分支的整体知识压缩能力有限

### 方向 2: 子树摘要优先

直接做某个分支下的树摘要，优先解决“知识沉淀”。

优点：

- 更符合 Thinking Tree 的知识树目标

缺点：

- 实现和维护复杂度更高

### 方向 3: 双层摘要

同时定义节点摘要和子树摘要，先落节点摘要，再逐步聚合成子树摘要。

优点：

- 最符合长期演进方向
- 便于后续分阶段落地

缺点：

- 设计阶段需要先把边界切清楚

## 建议的设计原则

1. summary 必须可消费，不能只是数据库里的一列空字段。
2. summary 必须有清晰粒度，不能混淆节点与子树职责。
3. summary 必须支持渐进落地，先可用，再变聪明。
4. summary 必须允许缺失，缺失时由 context-builder 明确降级。

## 开放问题

1. 本期 summary 先做节点摘要，还是直接定义节点 + 子树双层？
2. summary 生成应该同步还是异步？
3. summary 是否允许人工编辑，还是完全由模型生成？
4. summary 缺失时，是在 prompt 中占位，还是完全忽略？

## 建议的下一步

进入 `design-gate`，先把以下三件事定下来：

- summary 的粒度
- summary 的生成时机
- summary 的消费边界
